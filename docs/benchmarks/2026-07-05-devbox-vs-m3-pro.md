# Devbox vs M3 Pro Mac Agentic Coding Benchmark

Date: July 5, 2026

This page records the benchmark runs used to compare the lab KubeVirt devbox with a MacBook-class Apple M3 Pro workstation for local work that matters to terminal coding agents. The suite does not run Codex or Claude prompts. It measures local disk, filesystem, Git, search, validation, small edits, and concurrent command behavior.

## Machines and runs

| Run | Relevant spec | Repo commit | Result directory |
| --- | --- | --- | --- |

| Devbox after resize | KubeVirt VM, 8 vCPU, 32 GiB memory request, Ubuntu on ext4, 140 GiB Longhorn root disk | `13f335be8f9b` | `.cache/bench/results/20260705T101932Z-devbox-balanced` |
| Mac | Apple M3 Pro, 36 GB RAM, macOS 26.5.2, APFS on internal SSD | `ef4fa9c1de9d` | `.cache/bench/results/20260705T095309Z-aa-imc-g6gvmhkyht-balanced` |

The first devbox and Mac runs used the same repo commit and clean working trees. The resized devbox run happened after the benchmark docs and VM resource commits, so Mac comparisons against that run should be read as practical follow-up numbers rather than a perfectly identical commit comparison. The synthetic fixture tests are still directly useful because they are generated by the benchmark runner.

## Main result

The current 8 vCPU / 32 GiB devbox remains much faster than the Mac for agent-shaped command loops:

- Shared timed benchmarks: 20
- Devbox faster: 17
- Mac faster: 3
- Geometric mean: Mac command timings were about 2.84x slower than the resized devbox

The surprising part is that the Mac has much faster raw storage. The IO tests confirm that the Mac wins bulk sequential and random IO by 16x to 29x. The devbox wins the agentic coding shape because the workload is dominated by small files, metadata, Git, process startup, search, and copy/edit loops. Linux on ext4 handles this pattern much better than macOS/APFS in these runs.

## Current devbox vs Mac IO benchmarks

Higher is better. These numbers use the resized 8 vCPU / 32 GiB devbox run.

| Benchmark | Devbox 8/32 | Mac | Winner |
| --- | ---: | ---: | --- |
| Sequential write | 145 MiB/s | 3.3 GiB/s | Mac, 24.4x |
| Sequential read | 259 MiB/s | 5.4 GiB/s | Mac, 21.5x |
| Random mixed read | 6.7 MiB/s | 110 MiB/s | Mac, 16.4x |
| Random mixed write | 2.9 MiB/s | 47.5 MiB/s | Mac, 16.6x |
| fsync-heavy small writes | 1.4 MiB/s | 1.1 MiB/s | Devbox, 1.3x |

These numbers show the Longhorn-backed VM disk is much slower than the Mac SSD for raw throughput. The devbox result is still better for agent work because most agent loops do not stream multi-GiB files. They touch many small files and run many short commands.

## Effect of resizing the devbox

The devbox was resized from 4 vCPU / 16 GiB to 8 vCPU / 32 GiB and rebooted. Ubuntu then reported 8 CPUs and about 31 GiB of memory.

Across 20 shared timed benchmarks:

- New 8/32 devbox faster: 9
- Old 4/16 devbox faster: 6
- Similar: 5
- Geometric mean: the 8/32 run was about 1 percent faster overall

| Benchmark | Devbox 4/16 | Devbox 8/32 | Change |
| --- | ---: | ---: | --- |
| Concurrent agentic mix | 90.1 ms | 73.0 ms | 1.23x faster |
| Fixture search | 24.6 ms | 16.0 ms | 1.54x faster |
| Fixture Git init, add, commit | 525 ms | 506 ms | 1.04x faster |
| Fixture copy and edit | 494 ms | 480 ms | 1.03x faster |
| Repo local clone and status | 47.2 ms | 46.3 ms | Similar |
| Small-file create, stat, read, delete | 190 ms | 194 ms | Similar |
| Repo Git status | 2.1 ms | 2.9 ms | Slower in resized run |
| Repo Kustomize plus yq validate | 69.9 ms | 75.0 ms | Slower in resized run |

The resize is useful for comfort and concurrency, especially multiple agents, language servers, builds, and shells. It did not materially change the balanced benchmark because the measured workload is not mainly CPU or RAM bound.

## Storage locality note

After the resize, the VM was running on `talos-p13-7d3`, while the single Longhorn replica for the root disk was running on `talos-g6p-fik`. That means the VM disk path was still crossing the network.

This likely explains why raw IO barely changed after increasing CPU and memory. If the next goal is higher raw disk performance, storage placement or a different storage path is more likely to help than adding more vCPU.

## Reading the result

The right conclusion is not that the VM has faster storage. It does not. The result says that for this repo and synthetic agent-shaped workloads, the devbox is faster at the file and process patterns that coding agents use most often.

Likely contributors:

- Linux plus ext4 has low metadata overhead for many small files.
- macOS/APFS has higher small-file and metadata costs in this workload.
- macOS may also pay extra for file monitoring, Spotlight, endpoint security, and developer tooling hooks.
- The devbox is a headless server with fewer desktop background tasks.

The practical result is that the devbox is the better primary environment for Codex and Claude Code work in this lab. The 8/32 resize adds useful headroom, but the striking win was already present at 4/16.

## Notes

- The devbox Rust probe failed because the generated offline Cargo test did not complete successfully in both devbox runs.
- The Mac Rust probe was skipped because `cargo` was not installed.
- The Rust probe is excluded from the shared timing summaries above.
- Re-run the balanced profile after major macOS, Talos, KubeVirt, Longhorn, storage placement, or VM sizing changes.
