# Devbox vs M3 Pro Mac Agentic Coding Benchmark

Date: July 5, 2026

Visual companion: [HTML summary](2026-07-05-devbox-vs-m3-pro.html).

This page records the benchmark runs used to compare the lab KubeVirt devbox with a MacBook-class Apple M3 Pro workstation for local work that matters to terminal coding agents. The suite does not run Codex or Claude prompts. It measures local disk, filesystem, Git, search, validation, small edits, and concurrent command behavior.

The first devbox and Mac runs used the same repo commit and clean working trees. Later devbox runs happened after benchmark docs, VM resource, and storage class changes, so Mac comparisons against those runs should be read as practical follow-up numbers rather than perfectly identical commit comparisons. The synthetic fixture tests are still directly useful because they are generated by the benchmark runner.

## Main result

The current strict-local 8 vCPU / 32 GiB devbox remains much faster than the Mac for agent-shaped command loops:

- Shared timed benchmarks: 20
- Devbox faster: 17
- Mac faster: 3
- Geometric mean: Mac command timings remain about 2.8x slower than the current devbox

The command-loop conclusion did not change after recreating the VM with strict-local Longhorn storage. The new strict-local run is within about 1 percent of the closest local devbox command run by geometric mean.

The storage result did change. On the original post-resize run, the VM was still reading and writing a Longhorn replica on a different node. With a strict-local root disk, bulk and random IO are much faster than that cross-node result. The Mac still wins most raw IO throughput tests, but the gap is now much smaller, and the devbox is faster on fsync-heavy small writes.

The devbox wins the agentic coding shape because the workload is dominated by small files, metadata, Git, process startup, search, and copy/edit loops. Linux on ext4 handles this pattern much better than macOS/APFS in these runs.

## Current devbox vs Mac IO benchmarks

Higher is better. These numbers use the recreated strict-local 8 vCPU / 32 GiB devbox run. Mac values are rounded, so the ratios are approximate.

| Benchmark | Devbox strict-local | Mac | Winner |
| --- | ---: | ---: | --- |
| Sequential write | 262.6 MiB/s | 3.3 GiB/s | Mac, about 13x |
| Sequential read | 594.0 MiB/s | 5.4 GiB/s | Mac, about 9x |
| Random mixed read | 10.8 MiB/s | 110 MiB/s | Mac, about 10x |
| Random mixed write | 4.6 MiB/s | 47.5 MiB/s | Mac, about 10x |
| fsync-heavy small writes | 2.5 MiB/s | 1.1 MiB/s | Devbox, about 2.3x |

The strict-local Longhorn root disk is still much slower than the Mac internal SSD for bulk sequential and random mixed IO. It is no longer as far behind as the earlier cross-node Longhorn result, and it is now clearly ahead for fsync-heavy small writes. The devbox result is still better for agent work because most agent loops do not stream multi-GiB files. They touch many small files and run many short commands.

## Effect of resizing the devbox

The devbox was resized from 4 vCPU / 16 GiB to 8 vCPU / 32 GiB and rebooted. Ubuntu then reported 8 CPUs and about 31 GiB of memory.

Across 20 shared timed benchmarks:

- New 8/32 devbox faster: 9
- Old 4/16 devbox faster: 6
- Similar: 5
- Geometric mean: the 8/32 run was about 1 percent faster overall

| Benchmark | Devbox 4/16 | Devbox 8/32 | Change |
| --- | ---: | ---: | --- |
| Concurrent agentic mix | 90.1 ms | 73.0 ms | 1.23x faster |
| Fixture search | 24.6 ms | 16.0 ms | 1.54x faster |
| Fixture Git init, add, commit | 525 ms | 506 ms | 1.04x faster |
| Fixture copy and edit | 494 ms | 480 ms | 1.03x faster |
| Repo local clone and status | 47.2 ms | 46.3 ms | Similar |
| Small-file create, stat, read, delete | 190 ms | 194 ms | Similar |
| Repo Git status | 2.1 ms | 2.9 ms | Slower in resized run |
| Repo Kustomize plus yq validate | 69.9 ms | 75.0 ms | Slower in resized run |

The resize is useful for comfort and concurrency, especially multiple agents, language servers, builds, and shells. It did not materially change the balanced benchmark because the measured workload is not mainly CPU or RAM bound.

## Storage locality result

After the resize, the VM was running on `talos-p13-7d3`, while the original single Longhorn replica for the root disk was running on `talos-g6p-fik`. That means the VM disk path was crossing the network during the first 8 vCPU / 32 GiB benchmark run.

Longhorn rejected a direct switch to `strict-local` while the volume was attached. Switching temporarily to `best-effort` created a local replica on `talos-p13-7d3`; switching back to `disabled` left the single replica local to the VM node. A follow-up benchmark at `.cache/bench/results/20260705T105410Z-devbox-balanced` showed raw IO improvements before the VM was recreated.

The repo now manages the `longhorn-devbox-local` storage class, which provisions one strict-local replica on the same node as the VM. After recreating the VM with that storage class, the latest benchmark at `.cache/bench/results/20260705T125145Z-devbox-balanced` reported `OK: 25, skipped: 0, failed: 0`.

| Benchmark | Cross-node 8/32 | Manual local replica | Strict-local rebuild |
| --- | ---: | ---: | ---: |
| Sequential write | 145 MiB/s | 262 MiB/s | 262.6 MiB/s |
| Sequential read | 259 MiB/s | 626 MiB/s | 594.0 MiB/s |
| Random mixed read | 6.7 MiB/s | 10.0 MiB/s | 10.8 MiB/s |
| Random mixed write | 2.9 MiB/s | 4.3 MiB/s | 4.6 MiB/s |
| fsync-heavy small writes | 1.4 MiB/s | 2.0 MiB/s | 2.5 MiB/s |

The strict-local rebuild keeps the storage locality gains. Compared with the cross-node 8/32 run, it is about 1.6x to 2.3x higher on these IO tests. Compared with the manual local-replica follow-up, it is roughly the same on sequential write, 5 percent lower on sequential read, and better on random mixed IO and fsync-heavy small writes.

The local `.cache/bench/results/20260705T114216Z-devbox-balanced` run is another strict-local data point on the same commit. Compared with it, the latest `20260705T125145Z` run is 1.1 percent faster by timed-command geometric mean. Sequential write is 16 percent higher, fsync-heavy small writes are 12 percent higher, and the other IO tests are within about 3 percent.

## Reading the result

The right conclusion is not that the VM has a faster disk overall. It does not for bulk sequential or random mixed IO, even after strict-local Longhorn. The result says that for this repo and synthetic agent-shaped workloads, the devbox is faster at the file and process patterns that coding agents use most often.

Likely contributors:

- Linux plus ext4 has low metadata overhead for many small files.
- macOS/APFS has higher small-file and metadata costs in this workload.
- macOS may also pay extra for file monitoring, Spotlight, endpoint security, and developer tooling hooks.
- The devbox is a headless server with fewer desktop background tasks.
- Strict-local Longhorn removes the network hop from the VM root disk path.

The practical result is that the devbox is the better primary environment for Codex and Claude Code work in this lab. The 8/32 resize adds useful headroom, and the strict-local storage class makes the VM storage placement less fragile.
